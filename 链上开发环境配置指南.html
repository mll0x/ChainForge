1 概述与架构

本文档面向 Java 后端开发者，目标是从零搭建一个能发行 ERC-20 Token 和 ERC-721 NFT 的完整链上交互 Demo。技术栈以 Java + Solidity 为主，不引入不必要的语言。

为什么不用 Python + Rust？ 你是 Java 开发者，Web3j 是以太坊官方维护的 Java SDK，Spring Boot 集成成熟，无需额外学习新语言。Demo 阶段 Java + Solidity 完全够用。

语言职责分工

语言职责核心工具说明
Java后端服务 / 链交互Web3j + Spring Boot合约调用、交易签名、API 服务
Solidity智能合约Foundry / Hardhat链上逻辑：Token 标准、NFT 铸造
TypeScript前端（可选）React + Viem/WagmiDApp 用户界面、钱包连接

2 Web3 核心概念速览

作为 Java 开发者，以下概念可以类比你熟悉的后端体系：

Web3 概念简单解释Java 类比
区块链不可篡改的分布式账本只追加的分布式数据库
智能合约部署在链上自动执行的程序存储过程 / Service 层（无法修改已部署版本）
Gas执行链上操作的手续费云函数调用的计费单位
钱包管理私钥、签名交易的工具认证模块 + 密钥管理
RPC 节点区块链网络的读写入口数据库连接池 / API Gateway
ERC-20同质化代币标准（Fungible Token）积分系统 / 虚拟货币
ERC-721非同质化代币标准（NFT）唯一资产证书 / 数字产权
ERC-1155多代币标准（半同质化）批量资产管理
ABI合约接口描述（JSON）接口定义 / WSDL / OpenAPI
交易 Transaction链上状态变更操作数据库写事务（但确认需 12-30 秒）
事件 Event合约发出的日志消息队列 / 数据库触发器
测试网 Testnet模拟主网的免费测试环境Staging 环境
关键心智模型差异
  • 智能合约部署后不可修改，只能重新部署新版本（可使用代理模式升级）
  • 交易是异步确认的，不是即时生效
  • 所有链上操作都要花 Gas，读取数据免费
  • 私钥丢失 = 资产丢失，没有"找回密码"

5 ERC-20 Token 发行

5.1 ERC-20 标准接口

Solidity — ERC-20 核心方法
// 以下是你必须理解的核心方法
function totalSupply() external view returns (uint256);       // 总供应量
function balanceOf(address account) external view returns (uint256); // 查余额
function transfer(address to, uint256 amount) external returns (bool); // 转账
function approve(address spender, uint256 amount) external returns (bool); // 授权
function transferFrom(address from, address to, uint256 amount) external returns (bool); // 授权转账

5.2 使用 OpenZeppelin 编写 Token 合约

Solidity — src/MyToken.sol
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract MyToken is ERC20, Ownable {
    constructor(
        string memory name,       // "My Demo Token"
        string memory symbol,     // "MDT"
        uint256 initialSupply     // 1000000 (100万枚)
    ) ERC20(name, symbol) Ownable(msg.sender) {
        // mint 给部署者，decimals 默认 18
        _mint(msg.sender, initialSupply * 10 ** decimals());
    }

    // 只有 owner 能增发
    function mint(address to, uint256 amount) external onlyOwner {
        _mint(to, amount);
    }
}

6 ERC-721 NFT 铸造

6.1 ERC-721 核心概念

概念说明
tokenId每个 NFT 的唯一编号（uint256）
tokenURI指向 NFT 元数据的 URL（JSON：名称、描述、图片）
mint铸造新 NFT 的操作
ownerOf查询某个 tokenId 的持有者

6.2 NFT 元数据标准 (ERC-721 Metadata)

JSON — tokenURI 指向的内容
{
    "name": "My NFT #1",
    "description": "A demo NFT for learning Web3",
    "image": "ipfs://Qm.../image.png",
    "attributes": [
        { "trait_type": "Rarity", "value": "Common" },
        { "trait_type": "Power", "value": 42 }
    ]
}
元数据存储 图片和 JSON 文件通常上传到 IPFS（去中心化存储），通过 CID 引用。推荐使用 Pinata 或 nft.storage 免费托管。

6.3 编写 NFT 合约

Solidity — src/MyNFT.sol
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC721/ERC721.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract MyNFT is ERC721, Ownable {
    uint256 private _nextTokenId;
    string private _baseTokenURI;

    constructor(
        string memory name,       // "My Demo NFT"
        string memory symbol,     // "MDN"
        string memory baseURI     // "ipfs://Qm.../" 或 "https://api.example.com/metadata/"
    ) ERC721(name, symbol) Ownable(msg.sender) {
        _baseTokenURI = baseURI;
    }

    // 铸造 NFT 给指定地址
    function mint(address to) external onlyOwner returns (uint256) {
        uint256 tokenId = _nextTokenId++;
        _safeMint(to, tokenId);
        return tokenId;
    }

    // 批量铸造
    function batchMint(address to, uint256 quantity) external onlyOwner {
        for (uint256 i = 0; i < quantity; i++) {
            uint256 tokenId = _nextTokenId++;
            _safeMint(to, tokenId);
        }
    }

    function _baseURI() internal view override returns (string memory) {
        return _baseTokenURI;
    }

    function setBaseURI(string memory baseURI) external onlyOwner {
        _baseTokenURI = baseURI;
    }
}

6.4 ERC-721 vs ERC-1155

特性ERC-721ERC-1155
类型纯非同质化多代币（同质化 + 非同质化）
一次转账1 个 NFT多个不同类型代币
Gas较高批量操作更省 Gas
适用场景艺术品、收藏品游戏道具、半同质化票据
入门推荐先学这个理解 721 后再学

7.2 从 ABI 生成 Java 合约包装类

这是 Web3j 的核心工作流 用合约编译产出的 ABI + BIN 文件，自动生成 Java 类，之后在 Java 中像调用普通对象一样调用合约方法。
终端 — 使用 Web3j 命令行工具
# 安装 web3j CLI
brew install web3j

# 从 Foundry 编译产物生成 Java 类
web3j generate solidity \
    -a out/MyToken.sol/MyToken.abi.json \
    -b out/MyToken.sol/MyToken.bin \
    -o src/main/java \
    -p com.example.web3.contract

生成后在 com.example.web3.contract 包下会出现 MyToken.java，封装了所有合约方法的 Java 调用。

7.3 连接节点 & 查询余额

Java
@Service
public class Web3Service {

    private final Web3j web3j;

    public Web3Service() {
        // 连接 RPC 节点
        this.web3j = Web3j.build(new HttpService("https://eth-sepolia.g.alchemy.com/v2/YOUR_API_KEY"));
    }

    // 查询 ETH 余额
    public BigDecimal getEthBalance(String address) throws Exception {
        EthGetBalance balance = web3j.ethGetBalance(address, DefaultBlockParameterName.LATEST).send();
        return Convert.fromWei(balance.getBalance().toString(), Unit.ETHER);
    }

    // 查询 ERC-20 余额
    public BigInteger getErc20Balance(String contractAddress, String walletAddress) throws Exception {
        MyToken token = MyToken.load(
            contractAddress, web3j, Credentials.create("PRIVATE_KEY"),
            new DefaultGasProvider()
        );
        return token.balanceOf(walletAddress).send();
    }
}

7.4 调用合约 — 转账 Token

Java
public String transferToken(String contractAddress, String privateKey,
                            String to, BigInteger amount) throws Exception {

    Credentials credentials = Credentials.create(privateKey);
    MyToken token = MyToken.load(contractAddress, web3j, credentials, new DefaultGasProvider());

    // 调用合约的 transfer 方法
    TransactionReceipt receipt = token.transfer(to, amount).send();

    // 返回交易哈希
    return receipt.getTransactionHash();
}

7.5 调用合约 — 铸造 NFT

Java
public String mintNFT(String contractAddress, String privateKey,
                       String to) throws Exception {

    Credentials credentials = Credentials.create(privateKey);
    MyNFT nft = MyNFT.load(contractAddress, web3j, credentials, new DefaultGasProvider());

    TransactionReceipt receipt = nft.mint(to).send();
    return receipt.getTransactionHash();
}

9.3 后端签名 vs 前端签名

方式适用场景私钥位置
后端签名（Web3j）服务端自动化操作、批量铸造服务器环境变量 / KMS
前端签名（MetaMask）用户主动发起操作用户浏览器钱包
Demo 建议 先用后端签名（Web3j + 私钥在 .env）完成全流程，再考虑加前端 MetaMask 连接。

10 安全最佳实践

安全是链上开发的第一优先级 合约部署后无法修改，一个漏洞就可能导致资产永久丢失。Demo 阶段也要养成安全习惯。

10.1 私钥管理

  • 永远不要硬编码私钥在源代码中
  • 使用 .env 文件 + .gitignore
  • 生产环境使用 AWS KMS / HashiCorp Vault
  • 不同环境（开发/测试/生产）使用不同私钥
.env（不要提交到 Git！）
PRIVATE_KEY=0x...your_private_key
ALCHEMY_API_KEY=your_alchemy_key
ETHERSCAN_API_KEY=your_etherscan_key
.gitignore
.env
.env.local
*

10.2 合约安全检查清单

  • 使用 OpenZeppelin 合约库，不要自己写加密/权限逻辑
  • 所有外部调用检查重入攻击（OpenZeppelin 的 ReentrancyGuard）
  • 使用 require() 校验输入参数
  • 使用 Ownable / AccessControl 控制管理权限
  • 部署前用 forge test 覆盖所有边界情况
  • 考虑使用 Slither（静态分析）扫描合约
  • 主网部署前做专业审计

10.3 常见漏洞

漏洞说明防御
重入攻击外部调用后状态未更新ReentrancyGuard + Checks-Effects-Interactions
整数溢出Solidity 0.8+ 已内置检查使用 ^0.8.0 以上版本
权限缺失mint 等关键操作无权限控制Ownable / AccessControl
前端跑路合约 owner 可无限增发限制 maxSupply，或放弃所有权

14.3 AMM 核心数学：恒定乘积公式

x * y = k

这是 Uniswap V2 的核心公式，含义：

符号含义
x池中 Token A 的储备量 (reserve0)
y池中 Token B 的储备量 (reserve1)
k恒定乘积（交易前后 k 值不变，扣除手续费后略增）

兑换计算

公式
// 用 dx 个 Token A 换出 dy 个 Token B
// 扣除 0.3% 手续费后：实际输入 = dx * 997 / 1000
dy = y * (dx * 997) / (x * 1000 + dx * 997)

// 多跳兑换（通过 WETH 中转）
// TokenA → WETH → TokenB
amountOut1 = WETH_reserve * (amountIn * 997) / (TokenA_reserve * 1000 + amountIn * 997)
amountOut2 = TokenB_reserve * (amountOut1 * 997) / (WETH_reserve * 1000 + amountOut1 * 997)

添加流动性

公式
// 存入等价值的 Token A 和 Token B
// 铸造的 LP Token 数量 = min(
//   dx / reserve0 * totalSupply,
//   dy / reserve1 * totalSupply
// )
// 首次添加流动性：LP 数量 = sqrt(dx * dy)

14.4 WETH — 为什么需要包装 ETH？

核心原因 原生 ETH 不是 ERC-20 代币，没有 transfer()、approve()、balanceOf() 等标准方法。Uniswap 的 Pair 合约只处理 ERC-20 代币，所以需要先把 ETH 包装成 WETH。
操作ETH → WETHWETH → ETH
方法deposit()（存入 ETH，铸造等量 WETH）withdraw(uint256)（销毁 WETH，取回 ETH）
汇率1 ETH = 1 WETH（始终 1:1）1 WETH = 1 ETH
Gas~45,000 gas~30,000 gas
Solidity — WETH 核心逻辑
// WETH 合约极简实现（实际使用 OpenZeppelin 的 ERC20）
contract WETH {
    string public name = "Wrapped Ether";
    string public symbol = "WETH";
    uint8 public decimals = 18;

    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;

    function deposit() external payable {
        balanceOf[msg.sender] += msg.value;  // 存 ETH，铸造 WETH
        emit Deposit(msg.sender, msg.value);
    }

    function withdraw(uint256 amount) external {
        require(balanceOf[msg.sender] >= amount);
        balanceOf[msg.sender] -= amount;     // 销毁 WETH
        payable(msg.sender).transfer(amount); // 取回 ETH
        emit Withdrawal(msg.sender, amount);
    }
}

14.5 Router 合约详解

Router 是用户与 Pair 之间的中间层，处理复杂的路由逻辑：

方法功能关键参数
swapExactTokensForTokens用精确数量的输入 Token 换取尽可能多的输出 TokenamountIn, amountOutMin, path[], to, deadline
swapTokensForExactTokens用尽可能少的输入 Token 换取精确数量的输出 TokenamountOut, amountInMax, path[], to, deadline
swapExactETHForTokens用 ETH 换 Tokenmsg.value, amountOutMin, path[], to, deadline
swapExactTokensForETH用 Token 换 ETHamountIn, amountOutMin, path[], to, deadline
addLiquidity添加流动性tokenA, tokenB, amountADesired, amountBDesired, amountAMin, amountBMin, to, deadline
removeLiquidity移除流动性tokenA, tokenB, liquidity, amountAMin, amountBMin, to, deadline
path[] 路径参数 直接交易对：[TokenA, TokenB]。多跳：[TokenA, WETH, TokenB]。Router 会依次调用每个 Pair 的 swap 方法，实现链式兑换。

价格变化无常损失说明

15 第三阶段：EVM 底层与 MEV

理解 DeFi 之后，第三阶段深入 EVM 底层 —— 你的代码最终如何被虚拟机执行、Gas 如何计算、MEV 如何影响交易。这是从"会用"到"精通"的关键跨越。

为什么学这些？ 不理解 EVM，你无法写出真正省 Gas 的合约；不理解 MEV，你的交易可能被三明治攻击蚕食。这两个主题直接影响合约的安全性和用户资金安全。

15.1 核心概念全景

概念说明为什么重要
EVMEthereum Virtual Machine，执行智能合约的虚拟机所有合约最终编译成 EVM 字节码执行
Opcode操作码，EVM 的原子指令（如 ADD、SLOAD、CALL）理解 Gas 成本的根源，每个 Opcode 有固定 Gas 消耗
Gas 优化减少合约执行消耗的 Gas主网交易费用直接与 Gas 挂钩，优化 = 省钱
MEVMaximal Extractable Value，矿工/验证者可提取的最大价值你的交易可能被抢跑、三明治攻击

15.4 Gas 优化实战

优化 1：存储槽打包

Solidity — 优化前 vs 优化后
// ❌ 优化前：每个变量占一个槽位 → 3 次 SSTORE = ~15,000 Gas
struct UserInfo {
    uint256 balance;    // slot 0
    uint256 nonce;      // slot 1
    bool    isActive;   // slot 2
}

// ✅ 优化后：bool 和 uint128 打包到一个槽位 → 2 次 SSTORE = ~10,000 Gas
struct UserInfo {
    uint256 balance;      // slot 0
    uint128 nonce;        // slot 1 (低 128 位)
    bool    isActive;     // slot 1 (第 129 位)
    // nonce + isActive 共享一个 256 位槽位
}

优化 2：缓存存储变量

Solidity
// ❌ 每次循环都 SLOAD（冷访问 2100 Gas / 热访问 100 Gas）
for (uint256 i = 0; i < array.length; i++) { ... }

// ✅ 缓存到内存变量（3 Gas 读取）
uint256 len = array.length;  // 1 次 SLOAD
for (uint256 i = 0; i < len; i++) { ... }

优化 3：使用 unchecked

Solidity
// ❌ 默认溢出检查（每次 ++ 消耗额外 Gas）
for (uint256 i = 0; i < len; i++) { ... }

// ✅ 循环变量不可能溢出，跳过检查
for (uint256 i = 0; i < len;) {
    unchecked { ++i; }
}

优化 4：用事件替代存储

Solidity
// ❌ 存储到链上（SSTORE 5000+ Gas）
mapping(uint256 => string) public tokenURIs;

// ✅ 用事件记录（LOG ~1000 Gas，查询用 The Graph）
event TokenURISet(uint256 indexed tokenId, string uri);

// 链下通过 The Graph / 链上索引器查询历史事件

优化 5：Calldata 替代 Memory

Solidity
// ❌ 外部函数的 array 参数用 memory（复制一次）
function process(uint256[] memory items) external { ... }

// ✅ 只读时用 calldata（零拷贝，直接读交易输入）
function process(uint256[] calldata items) external { ... }

15.5 MEV 详解

什么是 MEV？

MEV (Maximal Extractable Value) 是区块生产者（矿工/验证者）通过重新排序、插入或删除交易来提取的额外利润。最初叫"Miner Extractable Value"，POS 后改为 "Maximal"。

常见 MEV 策略

策略原理影响防御
Front-running（抢跑）监控 mempool，发现有利交易后以更高 Gas 抢先执行用户多付钱私有交易池、Flashbots Protect
Back-running（尾随）在目标交易之后立即执行套利通常对用户无害—
Sandwich Attack（三明治）前买后卖：在目标交易前买入推高价格，目标交易执行后再卖出用户交易滑点增大设置严格 slippage、使用 Flashbots
Arbitrage（套利）不同 DEX 同一交易对价格不一致时，低买高卖促进价格收敛，正面作用—
Liquidation（清算）监控借贷协议，发现可清算仓位后抢先清算拿奖励协议正常运作的一部分—

三明治攻击图解

受害者提交 Swap：用 1000 USDC 换 ETH（设置 5% slippage） ↓ ┌─────────────────────────────────────────────────────┐ │ Mempool（待处理交易池） │ │ 攻击者看到这笔交易后： │ │ │ │ 1. Front-run：用更高 Gas 抢先买入 ETH → 价格上涨 │ │ 2. Victim：受害者的 Swap 执行 → 以更差价格成交 │ │ 3. Back-run：受害者成交后，攻击者卖出 ETH → 赚取差价 │ └─────────────────────────────────────────────────────┘ ↓ 攻击者利润 = 受害者的滑点损失

防御 MEV 的方法

方法原理适用场景
Flashbots Protect RPC交易不走公开 mempool，直接发给 Flashbots 拍卖所有用户，最简单
设置低 Slippage限制最大可接受滑点，三明治攻击无利可图小额交易，设 0.1%-0.5%
私有交易池通过 MEV Share / MEV Blocker 等提交大额交易
Batch Auction（批量拍卖）CoW Swap 等协议统一结算，消除排序优势DEX 交易
Commit-Reveal先提交哈希，再揭示真实操作，隐藏交易意图特定合约设计

16 常见问题

Q: 一定要用 Foundry 吗？能用 Hardhat 吗？
A: 都可以。Foundry 更快且用 Solidity 写测试（对 Java 开发者更直观），Hardhat 生态更成熟。Demo 阶段推荐 Foundry。
Q: Web3j 和 Ethers.js / Web3.py 有什么区别？
A: 功能相同，语言不同。Web3j = Java 版，Ethers.js = JavaScript 版，Web3.py = Python 版。Java 开发者用 Web3j 最自然。
Q: 测试网的 ETH 有价值吗？
A: 没有价值。测试网 ETH 是免费的，仅用于开发测试。主网 ETH 才有真实价值。
Q: 合约部署后能修改吗？
A: 不能直接修改。可以使用代理模式（OpenZeppelin 的 UUPS/Transparent Proxy）实现可升级合约，但增加了复杂度。Demo 阶段重新部署即可。
Q: Gas 费用为什么这么高？
A: 以太坊主网 Gas 较高。可以部署到 L2（Base、Arbitrum）降低 Gas，或使用测试网（免费）。Demo 阶段全程用测试网。
Q: Java 和 Rust 都要学吗？
A: Demo 阶段不需要 Rust。Java + Solidity 完全够用。如果后续需要高性能签名服务或自建节点，再考虑 Rust。
Q: 如何选择节点服务？
A: 开发阶段用 Alchemy 免费额度（30M CU/月）足够。生产环境建议付费或自建节点保证稳定性。
